The History of HTML

HTML, its supporting DOM APIs, as well as many of its supporting technologies, have been developed over a period of several decades. This has been done by a wide array of people with different priorities who, in many cases, did not know of each other's existence. Since the early days of the World Wide Web, there have been many versions of HTML:

Year Version
1989 | Tim Berners-Lee invented www.
1991 | Tim Berners-Lee invented HTML,
1993 | Dave Raggett drafted HTML+.
1995 | HTML Working Group defined HTML 2.0.
1997 | W3C Recommendation: HTML 3.2
1999 | W3C Recommendation: HTML 4.01.
2000 | W3C Recommendation: XHTML 1.0
2008 | WHATWG HTML5 First Public Draft
2012 | WHATWG HTML5 Living Standard
2014 | W3C Recommendation: HTML5
2016 | W3C Candidate Recommendation: HTML 5.1
2017 | W3C Recommendation: HTML5.1 2nd Edition
2017 | W3C Recommendation: HTML5.2

For its first five years, from 1990-1995, HTML went through a number of revisions and experienced a number of extensions. It was primarily hosted first at the European Organization for Nuclear Research (CERN) , and then at the Internet Engineering Task Force (IETF) . Then, with the creation of the World Wide Web Consortium (W3C), HTML's development changed venue again. A first abortive attempt at extending HTML in 1995 known as HTML 3.0 , then made way to a more pragmatic approach known as HTML 3.2 . This was completed in 1997, witch then saw HTML4 quickly follow later that same year.

The following year, the World Wide Web Consortium (W3C) membership decided to stop evolving HTML. And instead begin work on an XML-based equivalent, called XHTML. This effort started with a reformulation of HTML4 in XML, known as XHTML 1.0 . This added no new features except the new serialization, and was completed in 2000. After XHTML 1.0, the W3C's focus turned to making it easier for other working groups to extend XHTML, under the banner of XHTML Modularization. In parallel with this, the World Wide Web Consortium (W3C) also worked on a new language, not compatible with earlier HTML and XHTML languages, calling it XHTML2. Around the time that HTML's evolution was stopped in 1998, parts of the API for HTML developed by browser vendors were specified and published under the name DOM Level 1 in 1998. And then DOM Level 2 Core and DOM Level 2 HTML , starting in 2000 and then culminating in 2003. These efforts then petered out, with some DOM Level 3 specifications published in 2004, but the working group being closed before all the Level 3 drafts were completed.

In 2003, the publication of XForms, a technology positioned as the next generation of web forms, sparked renewed interest in evolving HTML itself (rather than finding replacements). This interest was from realization XML's deployment as a web technology, was limited to new technologies (like RSS and later Atom), rather than a replacement for existing technologies. A proof of concept to show it was possible to extend HTML4's forms to provide many of the features XForms 1.0 introduced, was the first result of this renewed interest. This was without requiring browsers to implement rendering engines that were incompatible with existing HTML web pages. At this early stage, while the draft was publicly available, and input already being solicited from all sources, the specification was only under Opera Software's copyright.

The idea that HTML's evolution should be reopened was tested at a World Wide Web Consortium (W3C) workshop in 2004. Here some principles that underlie HTML5 work, as well as the early draft proposal covering just forms-related features, were presented to the W3C jointly by Mozilla and Opera. The proposal was rejected on the grounds that the proposal conflicted with the previously chosen direction for the web's evolution. The World Wide Web Consortium (W3C) staff and membership, voted to continue developing XML-based replacements instead. Shortly thereafter, Apple, Mozilla and Opera, jointly announced intent to continue working on the effort under the umbrella of a new venue, called the Web Hypertext Application Technology Working Group (WHATWG) . A public mailing list was created, and the draft was moved to the Web Hypertext Application Technology Working Group (WHATWG) site. The copyright was subsequently amended to be jointly owned by all three vendors, and to allow reuse of the specification.

The WHATWG was based on several core principles, in particular that technologies need to be backwards compatible. Also, that specifications and implementations, need to match even if this means changing the specification rather than the implementations. And, that specifications need to be detailed enough that implementations, can achieve complete interoperability without reverse-engineering each other. The latter requirement in particular required the scope of the HTML5 specification, include what had previously been specified in three separate documents: HTML4, XHTML1 and DOM2 HTML. It also meant including significantly more detail than had previously been considered the norm.

In 2006, the World Wide Web Consortium (W3C) indicated an interest to participate in the development of HTML5 after all. And, in 2007 formed a working group chartered to work with the Web Hypertext Application Technology Working Group (WHATWG) on the development of the HTML5 specification. Apple, Mozilla and Opera allowed the W3C to publish the specification under the W3C copyright, while keeping a version with the less restrictive license on the WHATWG site.
For a number of years, both groups then worked together, in 2011, however, the groups came to the conclusion that they had different goals: The W3C wanted to publish a "finished" version of "HTML5". The WHATWG wanted to continue working on a Living Standard for HTML. And maintaining the specification rather than freezing it in a state with known problems, and adding new features as needed to evolve the platform.
In 2019, the WHATWG and W3C signed an agreement to collaborate on a single version of HTML going forward: The current specification.

HTML Character Sets - From ASCII to UTF-8

The HTML5 specification encourages web developers to use the UTF-8 character set, but this has not always been the case. Computer data is stored as binary codes (01000101) in the electronics. The character encoding for the early web was ASCII, and it was the first character encoding standard. To standardize the storing of text, the American Standard Code for Information Interchange (ASCII) was created. ASCII was the first character set (encoding standard) used between computers on the Internet. The character sets used in modern computers, in HTML, and on the Internet, are all based on ASCII. ASCII defined 128 different characters that could be used on the internet, it defined a unique binary number for each storable character. This was to support numbers from 0-9, the upper and lower case alphabet and special characters (like ! $ + - ( ) @ < > , .). Since ASCII used 7 bits for the character, it could only represent 128 different characters; it's biggest weakness however, was it excluded non English letters. ASCII is still in use today, especially in large mainframe computer systems - See the Complete ASCII Reference

Later, from HTML 2.0 to HTML 4.01, ISO-8859-1 was considered as the standard character set, and was the default character set for HTML 4. This character set supported 256 different character codes, and is an extension to ASCII; with added international characters. The International Standards Organization (ISO) defines the standard character sets, for different alphabets and languages - See the Complete ISO-8859-1 Reference
For Windows, ANSI (Windows-1252) was the original character set, and is identical to ISO-8859-1; but also has 32 extra characters. Windows-1252 was the default character set in Windows, up to Windows 95, and is an extension to ASCII; with added international characters. Historically, the term "ANSI Code Pages" was used in Windows to refer to non-DOS character sets. The intention was that these character sets would be ANSI standards like ISO-8859-1. It uses a full byte (8-bits) to represent 256 different characters, and has been the default in Windows; it is supported by all browsers. Even though Windows-1252 is almost identical to ISO-8859-1, it has never been an ANSI or ISO standard - See the Complete Windows-1252 Reference

With XML and HTML5, UTF-8 finally arrived and solved a lot of character encoding problems. The Unicode Consortium developed the UTF-8 and UTF-16 standards, because ISO-8859 character-sets are limited, and not a compatible multilingual environment. UTF-8 is also built on ASCII, the Unicode Standard covers almost all of the characters, punctuations and symbols in the world. All HTML 4 processors support UTF-8, and all HTML5 and XML processors support UTF-8, UTF-16, Windows-1252 and ISO-8859. The Unicode Standard has become a success and is implemented in HTML, XML, Java, JavaScript, E-mail, ASP, PHP, etc. The Unicode Consortium goal, is to replace the existing character sets with its standard Unicode Transformation Format (UTF). The Unicode standard is also supported in many operating systems and all modern browsers. The Unicode Consortium cooperates with the leading standards development organizations, like ISO, W3C and ECMA - See the Complete Unicode Reference
